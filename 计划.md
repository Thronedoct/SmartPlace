# SmartPlace 项目推进计划与协作流程

本文档是小组三人的执行手册，覆盖从当前 mock 阶段到最终答辩的先后关系、分工、联调流程、测试记录和材料流转。

当前结论：

- 项目选择课程方向 A：智能物体放置与合成图质量评价。
- 交付形态选择 Web 应用，不做 Android 端。
- `OPAAndroidDemoSimp/` 是老师提供的参考 Demo，可作为交互和素材参考，不纳入主线交付。
- 高分路线见 `docs/HIGH_SCORE_ROUTE.md`。
- 当前短期推进顺序见 `docs/NEXT_ROUTE.md`。

## 0. 总体路线

最终目标：

1. Web 工作台选择背景图和前景图。
2. Web 将图片上传到 FastAPI 后端。
3. 后端生成多个候选位置和尺度。
4. 后端合成候选图，调用 OPA/libcom 或改造后的评分模型。
5. 后端返回 Top 3 推荐位置、分数、标签、模型版本、耗时和可选解释图。
6. Web 展示推荐框、评分、候选切换、失败提示和结果证据字段。
7. 报告和 PPT 能说明模型改动、应用流程、结果验证、失败原因和 AI 辅助说明。

高分目标对应关系：

| 课程评分点 | 项目对应内容 | 主要负责人 |
|---|---|---|
| 应用目标与参考代码定位 | 方向 A Web 物体放置助手、OPA/libcom 说明 | A、B |
| 基础模型小改动 | `RGB -> RGB+mask` 输入改造、0-1 分数与三档标签 | A |
| 可交互应用 | Web 图片上传、参数控制、Top 3 推荐展示 | C |
| 测试案例与结果说明 | 18 组案例、成功/失败分析、模型对比表 | A、B、C |
| 报告、PPT 与分工 | 每人负责自己模块，C 统一材料整合 | C |
| 进阶项 | 模型解释、本地推理、复杂交互、候选排序 | A、B、C |

## 1. 默认分工

- 成员 A：模型改造与模型实验。
- 成员 B：后端推理服务、候选推荐管线和模型调用证据。
- 成员 C：Web 交互展示、演示视频、截图和最终材料整合。

每个成员不仅负责实现，也负责自己模块对应的报告和 PPT 内容。

## 2. GitHub 协作流程

仓库地址：

```text
https://github.com/Thronedoct/SmartPlace
```

### 2.1 分支约定

- `main`：稳定版本，只放已经能运行、能展示的内容。
- `dev`：日常集成分支，三个人的功能先合到这里。
- `feature/model-*`：模型实验、模型改造、推理脚本。
- `feature/backend-*`：接口、候选生成、推理服务、部署。
- `feature/web-*`：Web 页面、交互、候选展示、结果导出。
- `feature/docs-*`：报告、PPT、截图、计划文档等材料。

### 2.2 推荐工作流

1. 在 GitHub 上建 Issue，说明任务目标和验收标准。
2. 从最新 `dev` 拉分支。
3. 在自己的 feature 分支开发。
4. 本地运行最小检查。
5. 推送 feature 分支到 GitHub。
6. 开 Pull Request 到 `dev`。
7. 至少一名其他成员 review。
8. 修改通过后合并到 `dev`。
9. 阶段验收通过后，由一人把 `dev` 合并到 `main`。

### 2.3 PR 验收要点

- 后端 PR：`GET /api/health` 和 `POST /api/place/recommend` 能返回。
- Web PR：页面能打开，上传、提交、候选展示不崩溃。
- 模型 PR：脚本能运行，至少保存一份日志、截图或表格。
- 文档 PR：链接和路径正确，中文不乱码。
- 接口变化必须先更新 `docs/API.md`。

## 3. 接口协作流程

接口是三人协作的中心，Web 和后端必须以 `docs/API.md` 为准。

### 3.1 接口优先原则

如果要改接口字段，必须先做三件事：

1. 更新 `docs/API.md`。
2. 在 PR 或群里说明 Web 和后端分别要改哪里。
3. 保证 mock 后端、真实后端和 Web 解析逻辑同步更新。

### 3.2 阶段 0 Mock 流程

阶段 0 不接真实模型，只验证链路：

1. B 启动 `server/mock_stdlib.py` 或 FastAPI mock。
2. C 用 Web 工作台调用 `/api/place/recommend`。
3. 后端返回固定 Top 3。
4. Web 画出候选框、分数、模型版本和耗时。
5. A 根据返回结构确认模型输出需要适配哪些字段。

### 3.3 真实模型接入流程

真实模型接入时按下面顺序：

1. A 提供本地推理函数：输入合成图和 mask，输出分数。
2. B 在 FastAPI 中调用该函数。
3. B 保持返回 JSON 结构不变。
4. C 不改主界面，只展示真实返回字段。
5. 三人一起用一张新图片做端到端验证。

长期稳定字段：

- `request_id`
- `model_version`
- `coord_type`
- `runtime_ms`
- `best_index`
- `candidates`
- `rank`
- `x`
- `y`
- `w`
- `h`
- `score`
- `tier`
- `label`
- `preview_url`
- `heatmap_url`

## 4. 阶段 0：文档、mock 后端和 Web 工作台

目标：先把协作方式、仓库结构、接口和最小可跑 Web Demo 建起来。

共同任务：

- 建好 GitHub 仓库。
- 建好目录结构。
- 确认分支规则。
- 整理基础文档：
  - `README.md`
  - `docs/API.md`
  - `docs/HIGH_SCORE_ROUTE.md`
  - `docs/WORKFLOW.md`
  - `docs/TEST_CASES.md`
  - `docs/model_plan.md`
  - `docs/PHASE0_STATUS.md`
  - `计划.md`
- 统一接口字段。
- 明确三人分工和报告责任。

成员 A：

- 确定优先参考的模型代码：OPA、libcom 或 TopNet。
- 阅读参考模型输入输出说明。
- 在 `docs/model_plan.md` 中写清模型计划。
- 确认后端最终需要的模型输出格式，例如候选分数、排序和模型版本。

成员 B：

- 搭建 mock 服务。
- 实现 `GET /api/health`。
- 实现 `POST /api/place/recommend`。
- 返回固定 Top 3 候选位置和分数。
- 在 README 中写清运行方法。

成员 C：

- 维护 Web 工作台。
- 支持背景图、前景图、可选 mask 上传。
- 支持候选数量和前景尺度参数。
- 展示 Top 3 推荐框、分数、标签、耗时和模型版本。

阶段 0 验收：

- Web 能请求后端 mock 接口。
- 后端能返回固定推荐结果。
- Web 页面能显示推荐框、分数和 Top 3。
- README、接口文档和高分路线文档都存在。

## 5. 阶段 1：跑通参考模型与基础素材

目标：准备真实模型和测试素材，为替换 mock 做准备。

成员 A：

- 下载并跑通 OPA/libcom 参考示例。
- 记录运行环境、依赖、权重来源、输入输出格式。
- 整理最小推理脚本：输入合成图和 mask，输出合理性分数。
- 保存参考运行截图和终端日志。
- 在报告草稿中写“参考代码功能定位”。

成员 B：

- 根据 A 的推理脚本设计后端调用方式。
- 约定图片保存路径、临时文件命名和返回图片路径。
- 支持上传背景图和前景图，并保存到服务器或临时目录。
- 输出后端日志，记录请求文件名、候选数量、模型版本和耗时。

成员 C：

- 优化 Web 主流程。
- 准备主要页面状态：
  - 图片输入。
  - 推荐结果。
  - 候选详情。
  - 错误提示。
  - 运行证据字段。
- 用 mock 数据录一版早期演示视频。

共同任务：

- 收集第一批测试素材，覆盖桌面、地面、墙面、架子、户外和明显错误案例。
- 将案例记录到 `docs/TEST_CASES.md` 或共享表格。

阶段 1 验收：

- A 能单独跑通真实模型推理。
- B 能接收并保存 Web 上传图片。
- C 能完成 Web 图片输入和候选展示。
- 测试素材开始形成体系。

## 6. 阶段 2：打通真实端到端主链路

目标：把 mock 换成真实模型，让 Web 完成“上传 -> 后端推理 -> 返回结果 -> 展示”。

成员 A：

- 把推理脚本封装成稳定函数。
- 明确预处理流程：尺寸、通道、归一化和 mask 格式。
- 提供模型运行日志，方便 B 验证模型真实性。
- 确认模型输出能稳定转成 0 到 1 分数。

成员 B：

- 在 FastAPI 中接入 A 的模型推理函数。
- 实现候选位置生成。
- 实现边界过滤、尺度过滤和越界过滤。
- 对每个候选调用模型评分。
- 返回 Top 3 推荐结果。

成员 C：

- 展示真实返回的 Top 3 推荐框。
- 支持点击不同候选，切换预览图和评分。
- 增加加载状态和失败提示。
- 展示 `request_id`、`model_version`、`runtime_ms`。

阶段 2 验收：

- 能用一张新图片完成完整流程。
- 后端终端能看到真实模型推理日志。
- Web 能显示真实返回的 Top 3 结果。
- 这是中期汇报的最低可行版本。

## 7. 阶段 3：完成模型小改动

目标：满足三人组“至少两项模型相关工作，其中至少一项本体类改动”的要求。

成员 A：

- 完成本体类改动：将模型输入从 RGB 合成图改为 `RGB + mask`。
- 修改对应输入层或数据拼接逻辑。
- 保证改造后模型能真实推理。
- 将输出整理为 0 到 1 的合理性分数。
- 对比原模型和改造模型：
  - 排序结果。
  - Top 1 是否合理。
  - 推理时间。
  - 成功和失败案例。

成员 B：

- 支持原 RGB 模型和 `RGB+mask` 改造模型两种模式。
- 接口返回 `model_version`。
- 记录每次请求的推理耗时和候选数量。
- 保存推理日志作为答辩证据。

成员 C：

- 在 Web 结果区展示：
  - 分数。
  - 三档标签。
  - 推理耗时。
  - 当前模型版本。
- 保持界面清晰，优先服务课堂演示。

阶段 3 验收：

- 有明确模型修改文件和说明。
- 有原模型 vs 改造模型对比结果。
- Web 展示的是改造模型的真实输出。
- 报告中能讲清为什么这个改动服务于应用。

## 8. 阶段 4：进阶项与展示完整度

目标：让项目从“能跑”变成“好展示、可分析、能争取进阶分”。

优先进阶项：

- 模型改造：候选排序、分数校准、轻量 fine-tune 或 FOPA 候选对比。
- 本地推理：用本地电脑或局域网 FastAPI 服务真实推理，保留日志。
- 模型解释：Grad-CAM、遮挡实验或前景边界关注分析。
- 复杂交互：候选切换、参数调整、失败提示、结果导出或手动评分。

成员 A：

- 生成解释材料，例如 Grad-CAM、遮挡实验或前景边界关注分析。
- 为典型成功和失败案例各准备解释图。
- 整理解释图对应的简短分析。

成员 B：

- 生成 Top 3 候选预览图。
- 生成最佳推荐合成图。
- 返回可选热力图路径。
- 增加异常处理：
  - 图片格式错误。
  - 图片过大。
  - 模型推理失败。
  - 请求超时。

成员 C：

- 完成复杂交互：
  - Top 3 切换。
  - 参数调整。
  - 手动位置评分。
  - 结果导出。
  - 失败提示。
- 优化界面展示，让演示流程清楚顺滑。

阶段 4 验收：

- Web 不只是显示一个结果，而是能交互比较多个候选。
- 后端服务稳定，有错误提示。
- 有模型解释图和典型案例材料。

## 9. 阶段 5：系统测试与结果验证

目标：补齐高分项目最容易丢分的测试案例与结果说明。

成员 A：

- 负责模型实验表：
  - 原模型分数。
  - 改造模型分数。
  - 排序变化。
  - 推理时间。
  - 人工判断。
- 写模型失败原因分析。

成员 B：

- 负责接口和后端稳定性测试：
  - 不同图片尺寸。
  - 不同候选数量。
  - 连续请求。
  - 异常输入。
- 记录后端运行日志截图。

成员 C：

- 负责 Web 使用测试：
  - 演示录屏。
  - 正常流程。
  - 请求失败。
  - Top 3 切换。
  - 结果导出。
- 整理最终演示素材。

共同任务：

- 至少完成 18 组测试案例。
- 每组记录背景图、前景物体、推荐位置、分数、人工判断、是否成功和失败原因。
- 从 18 组中挑选 3 到 5 组放进 PPT 重点讲。

阶段 5 验收：

- 有成功案例。
- 有失败案例。
- 有表格。
- 有截图。
- 能说明系统什么时候可靠，什么时候容易失败。

## 10. 阶段 6：报告、PPT 与答辩准备

目标：把代码成果转成老师能看懂、能给分的材料。

成员 A：

- 负责参考代码定位。
- 负责模型输入输出。
- 负责 `RGB+mask` 改动说明。
- 负责模型对比表。
- 负责模型解释图。
- 负责失败案例分析。

成员 B：

- 负责整体架构图。
- 负责 Web 到后端的数据流。
- 负责 API 接口说明。
- 负责后端部署方式。
- 负责模型真实性证明。
- 负责后端日志截图。

成员 C：

- 负责应用目标。
- 负责用户流程。
- 负责 Web 界面截图。
- 负责演示录屏。
- 负责复杂交互说明。
- 负责分工表。
- 负责最终材料整合。

答辩演示流程：

1. 启动 FastAPI 后端。
2. 打开 Web 工作台。
3. 选择背景图。
4. 选择前景物体。
5. 点击自动推荐。
6. 后端终端显示收到请求和模型推理日志。
7. Web 展示 Top 3 推荐位置和分数。
8. 切换候选位置。
9. 展示一组模型解释图。
10. 展示一个失败案例并解释原因。

常见问答准备：

- 自己负责了哪些文件和功能？
- 项目如何证明真的调用了模型？
- 模型输入输出是什么？
- 为什么要加入 mask？
- Web 如何和后端通信？
- 如果请求失败怎么办？
- 哪些案例效果不好，原因是什么？
- AI 工具参与了哪些部分，小组如何核查和接管？

## 12. 最终提交清单

最终提交前检查：

- 代码已推到 GitHub。
- README 能说明如何运行。
- Web 应用有截图或录屏。
- 后端有运行截图和日志。

## 13. 推荐推进顺序

1. 整理文档，明确 Web 主线和高分路线。
2. 修复 Python/FastAPI 环境。
3. Web 和后端先用 mock 接口联调。
4. 跑通 OPA/libcom 参考模型。
5. 打通真实本地推理主链路。
6. 完成 `RGB+mask` 模型改造和候选排序。
7. 完善 Web 交互、解释图和后端稳定性。
8. 做测试案例、对比实验和失败分析。
9. 写报告、做 PPT、录屏并准备答辩。

最重要的节奏：每个阶段都先做一个可演示版本，再补细节和分析。
