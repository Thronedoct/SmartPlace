# SmartPlace 从零开始推进计划

本文档按先后关系拆解项目推进流程。默认三人分工如下：

- 成员 A：模型改造与模型实验。
- 成员 B：云端推理服务与候选推荐管线。
- 成员 C：Android App、交互展示和最终材料整合。

## 阶段 0：项目启动与最小 Demo

目标：先把协作方式、仓库结构、接口和最小可跑 Demo 建起来，避免后续各做各的。

### 共同任务

- 建立 Git 仓库和分支约定：
  - `main`：稳定交付版本。
  - `dev`：日常集成版本。
  - `feature/model-*`：模型相关工作。
  - `feature/backend-*`：后端和云端服务。
  - `feature/android-*`：Android App。
- 建立目录结构：
  - `OPAAndroidDemoSimp/`：课程 Android Demo 骨架。
  - `server/`：云端 FastAPI 或 mock 服务。
  - `docs/`：接口、流程、模型计划和测试记录。
  - `assets/`：测试图片和演示素材。
  - `report/`：报告、PPT、截图和录屏材料。
- 写好基础文档：
  - `README.md`：项目介绍和运行方式。
  - `docs/API.md`：前后端接口约定。
  - `docs/WORKFLOW.md`：协作流程和分支规则。
  - `docs/TEST_CASES.md`：测试案例记录模板。

### 成员 A：模型

- 确定优先参考的模型代码：OPA、libcom 或 TopNet。
- 在 `docs/model_plan.md` 中说明模型输入、输出和计划改动。
- 先确认后端最终需要的模型输出格式，例如候选分数、排序和模型版本。

### 成员 B：云端后端

- 搭建最小 mock 服务。
- 实现 `POST /api/place/recommend`。
- 阶段 0 先不跑真实模型，只返回固定 Top 3 候选位置和分数。
- 维护 `docs/API.md`，确保字段稳定。

### 成员 C：Android

- 导入课程 Android Demo 或建立自己的 Android 项目。
- 做最小页面：选择背景图、选择前景图、点击“推荐位置”。
- 先调用 mock 接口，能显示 Top 3 推荐框和评分即可。

### 阶段 0 验收

- Android 能请求后端 mock 接口。
- 后端能返回固定推荐结果。
- App 页面能显示推荐框、分数和 Top 3。
- 仓库、README 和接口文档已经存在。

## 阶段 1：跑通参考模型与基础素材

目标：准备真实模型和测试素材，为替换 mock 做准备。

### 成员 A：模型

- 下载并跑通 OPA/libcom 参考示例。
- 记录运行环境、依赖、权重来源、输入输出格式。
- 整理最小推理脚本：输入合成图，输出合理性分数。
- 保存参考运行截图和终端日志。

### 成员 B：后端

- 根据 A 的推理脚本设计后端调用方式。
- 约定图片保存路径、临时文件命名和返回图片路径。
- 支持上传背景图和前景图，并保存到服务器。

### 成员 C：Android

- 完成图片选择、图片预览和基础前景拖拽。
- 准备主要页面结构：图片输入页、推荐结果页、候选详情页和错误提示状态。

### 共同任务

- 收集第一批测试素材，覆盖桌面、地面、墙面、架子、户外和明显错误案例。

## 阶段 2：打通真实端到端主链路

目标：把 mock 换成真实模型，让 App 完成“上传 -> 云端推理 -> 返回结果 -> 展示”。

### 成员 A：模型

- 把推理脚本封装成稳定函数。
- 明确预处理流程：尺寸、通道、归一化和 mask 格式。
- 提供模型运行日志，方便 B 验证模型真实性。

### 成员 B：后端

- 在 FastAPI 中接入 A 的模型推理函数。
- 实现候选位置生成、边界过滤、尺度过滤和越界过滤。
- 对每个候选调用模型评分，并返回 Top 3。

### 成员 C：Android

- 把 mock 接口切换为真实云端接口。
- 展示 Top 3 推荐框。
- 支持点击不同候选，切换预览图和评分。
- 增加加载状态和失败提示。

## 阶段 3：完成模型小改动

目标：满足三人组“至少两项模型相关工作，其中至少一项本体类改动”的要求。

### 成员 A：模型

- 完成本体类改动：将模型输入从 RGB 合成图改为 `RGB + mask`。
- 修改对应输入层或数据拼接逻辑。
- 保证改造后模型能真实推理。
- 将输出整理为 0 到 1 的合理性分数。
- 对比原模型和改造模型的排序结果、Top 1 合理性、推理时间和失败案例。

### 成员 B：后端

- 支持原 RGB 模型和 `RGB+mask` 改造模型两种模式。
- 接口返回 `model_version`。
- 记录每次请求的推理耗时和候选数量。

### 成员 C：Android

- 在结果页展示分数、三档标签、推理耗时和当前模型版本。
- 保持界面清晰，优先服务课堂演示。

## 阶段 4：完善复杂交互与云端稳定性

目标：让项目从“能跑”变成“好展示、像一个完整应用”。

### 成员 A：模型

- 生成解释材料，例如 Grad-CAM、遮挡实验或前景边界关注分析。
- 为典型成功和失败案例各准备解释图。

### 成员 B：后端

- 生成 Top 3 候选预览图、最佳推荐合成图和可选热力图路径。
- 增加异常处理：图片格式错误、图片过大、模型推理失败和网络超时。
- 整理云端部署说明和运行截图。

### 成员 C：Android

- 完成复杂交互：前景拖拽、自动推荐、Top 3 切换、用户手动位置评分和结果导出。
- 优化界面展示，让演示流程清楚顺滑。

## 阶段 5：系统测试与结果验证

目标：补齐高分项目最容易丢分的测试案例与结果说明。

### 成员 A：模型

- 负责模型实验表：原模型分数、改造模型分数、排序变化、推理时间和人工判断。
- 写模型失败原因分析。

### 成员 B：后端

- 负责接口和云端稳定性测试：不同图片尺寸、不同候选数量、连续请求和异常输入。
- 记录云端运行日志截图。

### 成员 C：Android

- 负责 App 使用测试：真机录屏、正常流程、网络失败、Top 3 切换和手动拖拽评分。
- 整理最终演示素材。

### 共同任务

- 至少完成 18 组测试案例。
- 每组记录背景图、前景物体、推荐位置、分数、人工判断、是否成功和失败原因。

## 阶段 6：报告、PPT 与答辩准备

目标：把代码成果转成老师能看懂、能给分的材料。

### 成员 A：模型部分

- 负责参考代码定位、模型输入输出、`RGB+mask` 改动、模型对比、模型解释图和失败案例分析。

### 成员 B：系统部分

- 负责整体架构图、Android 到云端的数据流、API 接口、云端部署、模型真实性证明和后端日志截图。

### 成员 C：应用部分与总整合

- 负责应用目标、用户流程、Android 界面截图、演示录屏、复杂交互说明、分工表和最终材料整合。

## 推荐推进顺序

1. 建仓库、写文档、建立 mock 接口和 Android 小 Demo。
2. 跑通参考模型和图片上传。
3. 打通真实云端推理主链路。
4. 完成 `RGB+mask` 模型改造和候选排序。
5. 完善 App 交互、解释图和云端稳定性。
6. 做测试案例、对比实验和失败分析。
7. 写报告、做 PPT、录屏并准备答辩。

核心原则：先端到端跑通，再做模型改造，再补进阶项和展示材料。
