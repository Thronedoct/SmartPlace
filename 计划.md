# SmartPlace 项目推进计划与协作流程

本文档是小组三人的执行手册，覆盖从阶段 0 到最终答辩的先后关系、每个成员的任务、GitHub 协作方式、接口联调流程、测试记录和报告材料流转。

默认分工如下：

- 成员 A：模型改造与模型实验。
- 成员 B：云端推理服务与候选推荐管线。
- 成员 C：Android App、交互展示和最终材料整合。

核心原则：先打通端到端主链路，再做模型改造，再补进阶项和展示材料。任何阶段都要留下可运行代码、截图、日志或表格证据。

当前补充原则：先规划再行动。每轮开发前先更新对应文档中的目标、接口或验收标准；开发后必须同步测试结果、阶段状态和 GitHub 提交。

## 0. 总体路线

项目选择课程方向 A：智能物体放置与合成图质量评价。

最终目标：

1. Android App 选择背景图和前景图。
2. App 将图片上传到云端推理服务。
3. 后端生成多个候选位置。
4. 后端调用 OPA/libcom 或改造后的模型评分。
5. 后端返回 Top 3 推荐位置、分数、标签、耗时和可选解释图。
6. App 展示推荐框、评分、候选切换和结果导出。
7. 报告和 PPT 能说明模型改动、应用流程、结果验证和失败原因。

高分目标对应关系：

| 课程评分点 | 项目对应内容 | 主要负责人 |
|---|---|---|
| 应用目标与参考代码定位 | 方向 A 物体放置助手、OPA/libcom 说明 | A、B |
| 基础模型小改动 | `RGB -> RGB+mask` 输入改造、评分输出整理 | A |
| 可交互应用 | Android 图片选择、拖拽、Top 3 推荐展示 | C |
| 测试案例与结果说明 | 18 组案例、成功/失败分析、模型对比表 | A、B、C |
| 报告、PPT 与分工 | 每人负责自己模块，C 统一排版 | C |
| 进阶项 | 模型解释、复杂交互、多模型串联、云端推理 | A、B、C |

## 1. GitHub 协作流程

仓库地址：

```text
https://github.com/Thronedoct/SmartPlace
```

### 1.1 分支约定

- `main`：稳定版本，只放已经能运行、能展示的内容。
- `dev`：日常集成分支，三个人的功能先合到这里。
- `feature/model-*`：成员 A 的模型实验、模型改造、推理脚本。
- `feature/backend-*`：成员 B 的接口、候选生成、云端部署。
- `feature/android-*`：成员 C 的 Android 页面和交互。
- `feature/docs-*`：报告、PPT、截图、计划文档等材料。

分支命名示例：

```text
feature/model-rgb-mask
feature/backend-mock-api
feature/android-top3-display
feature/docs-phase0-plan
```

### 1.2 推荐工作流

每个任务都按下面顺序走：

1. 在 GitHub 上建 Issue，说明任务目标和验收标准。
2. 从最新 `dev` 拉分支。
3. 在自己的 feature 分支开发。
4. 本地运行最小检查。
5. 推送 feature 分支到 GitHub。
6. 开 Pull Request 到 `dev`。
7. 至少一名其他成员 review。
8. 修改通过后合并到 `dev`。
9. 每个阶段验收通过后，由一人把 `dev` 合并到 `main`。

常用命令：

```bash
git checkout main
git pull
git checkout -b dev
git push -u origin dev

git checkout dev
git pull
git checkout -b feature/backend-mock-api

git add .
git commit -m "Add mock recommendation API"
git push -u origin feature/backend-mock-api
```

### 1.3 Issue 使用规则

每个 Issue 尽量写清：

- 背景：为什么要做。
- 任务：具体改什么。
- 负责人：A、B 或 C。
- 验收：做到什么算完成。
- 证据：截图、日志、表格或录屏放在哪里。

Issue 标题示例：

```text
[Backend] 接入 mock 推荐接口
[Android] 展示 Top 3 推荐框
[Model] 跑通 OPA 参考推理
[Docs] 整理阶段 0 汇报材料
```

### 1.4 PR 规则

每个 PR 至少包含：

- 改了什么。
- 为什么改。
- 怎么验证。
- 是否影响接口。
- 是否需要其他成员同步修改。

PR 合并前检查：

- 后端 PR：`GET /api/health` 和 `POST /api/place/recommend` 能返回。
- Android PR：页面能打开，关键按钮不崩溃。
- 模型 PR：脚本能运行，至少保存一份日志或截图。
- 文档 PR：链接和路径正确，中文不乱码。

### 1.5 Commit 规范

提交信息用简短英文或中文都可以，但要能看懂。

推荐格式：

```text
Add mock recommendation API
Update Android result display
Document RGB mask model plan
修复接口字段说明
```

不推荐：

```text
update
fix
123
乱改一下
```

### 1.6 冲突处理

如果发生冲突：

1. 先停止继续写新代码。
2. 找到冲突文件负责人。
3. 由该文件负责人决定保留哪部分。
4. 解决后在 PR 中说明冲突原因。

不要直接覆盖别人的改动，尤其是 `docs/API.md`、Android 主页面和后端接口文件。

## 2. 接口协作流程

接口是三人协作的中心。Android 和后端必须以 `docs/API.md` 为准。

### 2.1 接口优先原则

如果要改接口字段，必须先做三件事：

1. 更新 `docs/API.md`。
2. 在 PR 或群里说明 Android 和后端分别要改哪里。
3. 保证 mock 后端和 Android 解析逻辑同步更新。

### 2.2 阶段 0 Mock 流程

阶段 0 不接真实模型，只验证链路：

1. B 启动 `server/mock_stdlib.py`。
2. C 用 Android 调用 `/api/place/recommend`。
3. 后端返回固定 Top 3。
4. Android 画出候选框和分数。
5. A 根据返回结构确认模型输出需要适配哪些字段。

### 2.3 真实模型接入流程

真实模型接入时按下面顺序：

1. A 提供本地推理函数：输入图片和候选框，输出分数。
2. B 在 FastAPI 中调用该函数。
3. B 保持返回 JSON 结构不变。
4. C 不改 UI，只把 mock 地址切换到真实地址。
5. 三人一起用一张新图片做端到端验证。

### 2.4 接口字段稳定要求

尽量长期保留这些字段：

- `request_id`
- `model_version`
- `coord_type`
- `runtime_ms`
- `best_index`
- `candidates`
- `rank`
- `x`
- `y`
- `w`
- `h`
- `score`
- `tier`
- `label`
- `preview_url`
- `heatmap_url`

后续增加字段可以，但不要随便删除或改名。

## 3. 日常同步流程

每次同步建议控制在 15 到 20 分钟，按固定顺序讲：

1. A 说模型进度：参考代码、推理脚本、模型改动、实验结果。
2. B 说后端进度：接口、部署、日志、候选排序、联调问题。
3. C 说 App 进度：页面、交互、录屏、展示问题。
4. 一起确认接口是否变化。
5. 更新 Issue 状态和下一步负责人。

每次同步后至少留下一个记录：

- GitHub Issue 评论。
- `docs/PHASE0_STATUS.md` 或阶段状态文档。
- 测试表格截图。
- PR 说明。

## 4. 阶段 0：项目启动与最小 Demo

目标：先把协作方式、仓库结构、接口和最小可跑 Demo 建起来，避免后续各做各的。

### 共同任务

- 建好 GitHub 仓库。
- 建好目录结构。
- 确认分支规则。
- 写好基础文档：
  - `README.md`
  - `docs/API.md`
  - `docs/WORKFLOW.md`
  - `docs/TEST_CASES.md`
  - `docs/model_plan.md`
  - `计划.md`
- 统一接口字段。
- 明确三人分工和报告责任。

### 成员 A：模型

- 确定优先参考的模型代码：OPA、libcom 或 TopNet。
- 阅读参考模型输入输出说明。
- 在 `docs/model_plan.md` 中写清模型计划。
- 从课程 PDF 中提取并浏览方向 A 的参考 GitHub 源码，记录到 `docs/model_plan.md`。
- 确认后端最终需要的模型输出格式，例如候选分数、排序和模型版本。

### 成员 B：云端后端

- 搭建 mock 服务。
- 实现 `GET /api/health`。
- 实现 `POST /api/place/recommend`。
- 返回固定 Top 3 候选位置和分数。
- 在 README 中写清运行方法。

### 成员 C：Android

- 导入课程 Android Demo 或建立自己的 Android 项目。
- 如果重建 Android 项目，需要先写清目录名、技术栈和仍然遵守的 API 字段。
- 做最小页面：选择背景图、选择前景图、点击“推荐位置”。
- 先调用 mock 接口。
- 能显示 Top 3 推荐框和评分。

### 阶段 0 验收

- GitHub 上已有项目代码。
- Android 能请求后端 mock 接口。
- 后端能返回固定推荐结果。
- App 页面能显示推荐框、分数和 Top 3。
- README、接口文档和计划文档都存在。

## 5. 阶段 1：跑通参考模型与基础素材

目标：准备真实模型和测试素材，为替换 mock 做准备。

### 成员 A：模型

- 下载并跑通 OPA/libcom 参考示例。
- 记录运行环境、依赖、权重来源、输入输出格式。
- 整理最小推理脚本：输入合成图，输出合理性分数。
- 保存参考运行截图和终端日志。
- 在报告草稿中写“参考代码功能定位”。

### 成员 B：后端

- 根据 A 的推理脚本设计后端调用方式。
- 约定图片保存路径、临时文件命名和返回图片路径。
- 支持上传背景图和前景图，并保存到服务器。
- 输出后端日志，记录请求文件名、候选数量和耗时。

### 成员 C：Android

- 完成图片选择、图片预览和基础前景拖拽。
- 准备主要页面结构：
  - 图片输入页。
  - 推荐结果页。
  - 候选详情页。
  - 错误提示状态。
- 用 mock 数据录一版早期演示视频。

### 共同任务

- 收集第一批测试素材，覆盖桌面、地面、墙面、架子、户外和明显错误案例。
- 将案例记录到 `docs/TEST_CASES.md` 或共享表格。

### 阶段 1 验收

- A 能单独跑通真实模型推理。
- B 能接收并保存 Android 上传图片。
- C 能完成图片导入和基础拖拽。
- 测试素材开始形成体系。

## 6. 阶段 2：打通真实端到端主链路

目标：把 mock 换成真实模型，让 App 完成“上传 -> 云端推理 -> 返回结果 -> 展示”。

### 成员 A：模型

- 把推理脚本封装成稳定函数。
- 明确预处理流程：尺寸、通道、归一化和 mask 格式。
- 提供模型运行日志，方便 B 验证模型真实性。
- 确认模型输出能稳定转成 0 到 1 分数。

### 成员 B：后端

- 在 FastAPI 中接入 A 的模型推理函数。
- 实现候选位置生成。
- 实现边界过滤、尺度过滤和越界过滤。
- 对每个候选调用模型评分。
- 返回 Top 3 推荐结果。

### 成员 C：Android

- 把 mock 接口切换为真实云端接口。
- 展示 Top 3 推荐框。
- 支持点击不同候选，切换预览图和评分。
- 增加加载状态和失败提示。

### 阶段 2 验收

- 能用一张新图片完成完整流程。
- 后端终端能看到真实模型推理日志。
- App 能显示真实返回的 Top 3 结果。
- 这是中期汇报的最低可行版本。

## 7. 阶段 3：完成模型小改动

目标：满足三人组“至少两项模型相关工作，其中至少一项本体类改动”的要求。

### 成员 A：模型

- 完成本体类改动：将模型输入从 RGB 合成图改为 `RGB + mask`。
- 修改对应输入层或数据拼接逻辑。
- 保证改造后模型能真实推理。
- 将输出整理为 0 到 1 的合理性分数。
- 对比原模型和改造模型：
  - 排序结果。
  - Top 1 是否合理。
  - 推理时间。
  - 成功和失败案例。

### 成员 B：后端

- 支持原 RGB 模型和 `RGB+mask` 改造模型两种模式。
- 接口返回 `model_version`。
- 记录每次请求的推理耗时和候选数量。
- 保存推理日志作为答辩证据。

### 成员 C：Android

- 在结果页展示：
  - 分数。
  - 三档标签。
  - 推理耗时。
  - 当前模型版本。
- 保持界面清晰，优先服务课堂演示。

### 阶段 3 验收

- 有明确模型修改文件和说明。
- 有原模型 vs 改造模型对比结果。
- App 展示的是改造模型的真实输出。
- 报告中能讲清为什么这个改动服务于应用。

## 8. 阶段 4：完善复杂交互与云端稳定性

目标：让项目从“能跑”变成“好展示、像一个完整应用”。

### 成员 A：模型

- 生成解释材料，例如 Grad-CAM、遮挡实验或前景边界关注分析。
- 为典型成功和失败案例各准备解释图。
- 整理解释图对应的简短分析。

### 成员 B：后端

- 生成 Top 3 候选预览图。
- 生成最佳推荐合成图。
- 返回可选热力图路径。
- 增加异常处理：
  - 图片格式错误。
  - 图片过大。
  - 模型推理失败。
  - 网络请求超时。
- 整理云端部署说明和运行截图。

### 成员 C：Android

- 完成复杂交互：
  - 前景拖拽。
  - 自动推荐。
  - Top 3 切换。
  - 用户手动位置评分。
  - 结果导出。
- 优化界面展示，让演示流程清楚顺滑。

### 阶段 4 验收

- App 不只是显示一个结果，而是能交互比较多个候选。
- 后端服务稳定，有错误提示。
- 有模型解释图和典型案例材料。

## 9. 阶段 5：系统测试与结果验证

目标：补齐高分项目最容易丢分的测试案例与结果说明。

### 成员 A：模型

- 负责模型实验表：
  - 原模型分数。
  - 改造模型分数。
  - 排序变化。
  - 推理时间。
  - 人工判断。
- 写模型失败原因分析。

### 成员 B：后端

- 负责接口和云端稳定性测试：
  - 不同图片尺寸。
  - 不同候选数量。
  - 连续请求。
  - 异常输入。
- 记录云端运行日志截图。

### 成员 C：Android

- 负责 App 使用测试：
  - 真机录屏。
  - 正常流程。
  - 网络失败。
  - Top 3 切换。
  - 手动拖拽评分。
- 整理最终演示素材。

### 共同任务

- 至少完成 18 组测试案例。
- 每组记录背景图、前景物体、推荐位置、分数、人工判断、是否成功和失败原因。
- 从 18 组中挑选 3 到 5 组放进 PPT 重点讲。

### 阶段 5 验收

- 有成功案例。
- 有失败案例。
- 有表格。
- 有截图。
- 能说明系统什么时候可靠，什么时候容易失败。

## 10. 阶段 6：报告、PPT 与答辩准备

目标：把代码成果转成老师能看懂、能给分的材料。

### 成员 A：模型部分

- 负责模型输入输出。
- 负责 `RGB+mask` 改动说明。
- 负责模型对比表。
- 负责模型解释图。
- 负责失败案例分析。

### 成员 B：系统部分

- 负责整体架构图。
- 负责 Android 到云端的数据流。
- 负责 API 接口说明。
- 负责云端部署方式。
- 负责模型真实性证明。
- 负责后端日志截图。

### 成员 C：应用部分与总整合

- 负责应用目标。
- 负责用户流程。
- 负责 Android 界面截图。
- 负责演示录屏。
- 负责复杂交互说明。
- 负责分工表。
- 负责最终材料整合。

### 答辩演示流程

建议按固定脚本演示：

1. 打开 Android App。
2. 选择背景图。
3. 选择前景物体。
4. 点击自动推荐。
5. 后端终端显示收到请求和模型推理日志。
6. App 展示 Top 3 推荐位置和分数。
7. 切换候选位置。
8. 展示一组模型解释图。
9. 展示一个失败案例并解释原因。

### 常见问答准备

每个人都要能回答：

- 自己负责了哪些文件和功能？
- 项目如何证明真的调用了模型？
- 模型输入输出是什么？
- 为什么要加入 mask？
- App 如何和云端通信？
- 如果网络失败怎么办？
- 哪些案例效果不好，原因是什么？

## 13. 推荐推进顺序

1. 建仓库、写文档、建立 mock 接口和 Android 小 Demo。
2. 建 GitHub Issue，按三条主线分配任务。
3. Android 和后端先用 mock 接口联调。
4. 跑通参考模型和图片上传。
